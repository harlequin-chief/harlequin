# RESULTS — Woven Trust Consensus simulator
> Generated by `run_consensus.py` (stdlib only). Reproducible (seeded PRNG). Sub-sampled Snowball/Avalanche voting with **reputation-weighted** sampling (PAPER §5.4).
> Parameters: sample k=20, quorum α=14, decision β=12, 60 runs per point. Honest nodes start at the legitimate value; the adversary is Byzantine (pushes a conflicting value).

## 1. Security threshold by the adversary's REPUTATION FRACTION
Few adversarial nodes, but controlling a growing fraction of the total reputation.

| Adversarial reputation | Safe consensus (everyone the legitimate value) | Capture (some honest flipped) | Fork |
|---|---|---|---|
| 0 % | 100 % | 0 % | 0 % |
| 10 % | 100 % | 0 % | 0 % |
| 20 % | 50 % | 0 % | 0 % |
| 30 % | 0 % | 0 % | 0 % |
| 40 % | 0 % | 97 % | 0 % |
| 50 % | 0 % | 100 % | 0 % |

**Reading (distinguishes two things):**
- **Safety** (no honest accepts the false value, no fork): holds while the adversary has **less than ~40 %** of the reputation. Below that threshold the adversary **never gets a false decision**.
- **Liveness** (everyone reaches a decision): starts degrading earlier, around **~20 %**: there the adversary still **fools nobody**, but can **stall** convergence (honest nodes left undecided). A jam, not a deception.
- The key point: both thresholds are expressed as a **fraction of reputation**, not a number of nodes. That is the thesis.

## 2. Fake crowd (Sybil): number of nodes vs reputation
The adversary has **1000 nodes** (93 % of the network) with reputation ~0; the honest nodes are a minority of nodes but hold all the reputation.

| Sampling | Safe consensus | Capture | Fork |
|---|---|---|---|
| **Reputation-weighted** (WTC) | 100 % | 0 % | 0 % |
| Uniform (plain Avalanche, contrast) | 0 % | 100 % | 0 % |

**Reading:** with reputation-weighted sampling, a fake crowd of 93 % of the nodes does **not break** the network (still safe), because it almost never enters the samples: power is reputation, not number (Art. VI). With **uniform** sampling — no weighting — the same crowd captures/splits the network. Reputation-weighting is what defends (parallel to the anti-collusion damping of the engine).

## 3. INDEPENDENCE-weighted sampling vs a correlated adversary (PAPER §5.4)
Now the adversary is subtler: not a crowd without reputation, but a **bloc that DID earn reputation**, all of it **correlated** (one trust cluster vouching among itself). Reputation-only sampling treats it as independent. WTC's defence: weight also by **independence**, capping at `cap` the nodes a single cluster can contribute to each committee of k=20 (α=14).

| Adversarial reputation (1 cluster) | rep-only: safe / capture | +independence (cap=3): safe / capture |
|---|---|---|
| 30 % | 0 % / 0 % | 100 % / 0 % |
| 40 % | 0 % / 97 % | 100 % / 0 % |
| 45 % | 0 % / 100 % | 100 % / 0 % |
| 50 % | 0 % / 100 % | 100 % / 0 % |

**Reading:** with rep-only sampling, a correlated bloc with ≥~40 % of the reputation **captures** the network. Independence-weighted sampling (per-cluster cap) **neutralises** it: the bloc cannot occupy enough committee seats even with the reputation. Structurally, to force the false colour it needs ⌈α/cap⌉ seats from distinct clusters.

**Honest frontier — the adversary fragments its bloc** (45 % reputation, cap=3):

| Adversary clusters | +independence (cap=3): safe / capture |
|---|---|
| 1 | 100 % / 0 % |
| 2 | 100 % / 0 % |
| 3 | 0 % / 0 % |
| 4 | 0 % / 0 % |
| 5 | 0 % / 93 % |
| 6 | 0 % / 100 % |

The protection holds until the adversary fragments into **⌈α/cap⌉ = ⌈14/3⌉ = 5** distinct clusters; with 5+ it captures again. But fragmenting requires **each** sub-bloc to look like an **independent** cluster to community detection — exactly what the **reputation engine** (damping + communities) is built to resist, and which we already showed makes laundering yield **0** consensus power. *The two prototypes compose:* the consensus's independence sampling rests on the cluster identification the engine provides.

## 4. ADAPTIVE adversary (anti-finality splitter)
Instead of always pushing the same value, the adversary each round reports the **minority** colour among the honest to keep them split and stop any colour from settling (worst-case: it sees the current state). Measures whether it can break SAFETY or only LIVENESS.

| Adversarial reputation (1 cluster) | adaptive, rep-only: safe / capture / stall | adaptive, +indep cap=3 |
|---|---|---|
| 20 % | 50 % / 0 % / 50 % | 100 % / 0 % / 0 % |
| 30 % | 0 % / 0 % / 100 % | 100 % / 0 % / 0 % |
| 40 % | 0 % / 0 % / 100 % | 100 % / 0 % / 0 % |
| 50 % | 0 % / 0 % / 100 % | 100 % / 0 % / 0 % |

**Reading:** the adaptive adversary **never forces a false decision** (capture 0 %): it attacks **liveness** (stalls convergence), not **safety**. And under independence-weighted sampling it cannot even stall: its correlated bloc does not enter the committees enough. Consistent with Snowball/Avalanche: safety is robust; the adaptive cost is liveness, recoverable.

## 5. Network partition: partition attack + network-quorum mitigation
Drops the complete-network assumption: the honest nodes split into a large group A (60) and a small group B (20); a globally harmless adversary (15%) concentrates its reputation in B. The network is partitioned `D` rounds and then heals.

| D (partition rounds) | fork (no mitig.) | fork (+quorum 60%) | stall (+quorum) |
|---:|---:|---:|---:|
| 0 | 0 % | 0 % | 0 % |
| 20 | 40 % | 0 % | 0 % |
| 50 | 97 % | 2 % | 0 % |
| 90 | 100 % | 2 % | 33 % |

**Finding (honest, important):** a long partition **concentrates the adversary's LOCAL share** in the small group above the threshold → B finalises the false value → on heal, **FORK**. A 15% global harmless adversary becomes **100% fork** with a 90-round partition. **A real safety failure**, found in simulation before building the chain.

**Mitigation:** condition **finality** on seeing a **quorum of the total reputation** (a node does not decide if it reaches <60% of the network). Under partition, B (25% of the reputation) does not reach quorum → it **does not finalise, it stalls** (liveness cost) instead of forking, and **recovers on heal**. Fork drops from ~100% to ~0–2%. The expected rule of a robust BFT: **when in doubt about a partition, prefer to halt (liveness) over deciding wrongly (safety)**. Pending: dynamic detection of the seen fraction (modelled here with the group) and latency/message loss.

## 6. Message loss: latency / unreliable network
Each queried response is lost with probability `p` (models latency and network loss). Measures whether the unreliable network breaks safety or only slows it.

| loss p | no adversary: safe / stall | adversary 25%: capture / stall |
|---:|---:|---:|
| 0 % | 100 % / 0 % | 0 % / 100 % |
| 20 % | 53 % / 47 % | 0 % / 100 % |
| 40 % | 0 % / 100 % | 0 % / 100 % |
| 60 % | 0 % / 100 % | 0 % / 100 % |

**Reading:** loss degrades **liveness** (more stalls) but **never breaks safety** (capture 0 % at any loss): the threshold α does not change. Parameter implication: to make progress under loss `p` you need enough live responses, **α ≤ k·(1−p)**; with k=20, α=14 it tolerates up to ~30 % loss before stalling completely. The α/k choice bounds the tolerable loss — a parameter to set with real network modelling.

## Conclusion
The simulator confirms and hardens the paper's pieces: (1) the **security threshold is measured in a fraction of REPUTATION**, not of nodes; (2) **independence-weighted sampling** (PAPER §5.4) raises that threshold against a **correlated** adversary — beating it requires fragmenting into clusters that look independent, which the reputation engine resists (*the two prototypes compose*); (3) an **adaptive** adversary only attacks **liveness**, never safety; (4) under a **partition** a real fork appears if finality is reached blindly, **mitigated** by conditioning finality on a network quorum (safety over liveness); (5) **message loss** only slows liveness, never safety. Remaining honest limitation: a binary-decision model and a **formal** safety/liveness proof (PAPER §10).
